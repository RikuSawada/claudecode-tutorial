# Issue 作成ガイドライン

Claude Code が GitHub Issue を作成・更新する際に従うルール。

---

## 基本方針

- Issue は「何を・なぜ・どう解決するか」が一目でわかるように書く
- 1 Issue = 1 トピック。複数の変更を混在させない
- 作成後は必ず `gh issue view <番号>` で内容を確認する

---

## タイトルのルール

| 種別 | プレフィックス | 例 |
|------|--------------|-----|
| バグ | `[Bug]` | `[Bug] タスク削除後に一覧が更新されない` |
| 機能追加 | `[Feature]` | `[Feature] タスクにラベルを付けられるようにする` |
| 改善 | `[Improve]` | `[Improve] タスク一覧の読み込みを高速化する` |
| ドキュメント | `[Docs]` | `[Docs] API エンドポイント一覧を README に追加する` |
| リファクタリング | `[Refactor]` | `[Refactor] crud/tasks.py の重複ロジックを整理する` |

---

## 本文テンプレート

### バグ報告

```markdown
## 概要
（バグの内容を 1〜2 文で説明）

## 再現手順
1. 〇〇ページを開く
2. 〇〇ボタンをクリック
3. 〇〇が発生する

## 期待する動作
（本来どう動くべきか）

## 実際の動作
（実際に起きていること。エラーメッセージがあれば貼る）

## 環境
- OS:
- Python:
- ブラウザ:
```

### 機能追加・改善

```markdown
## 概要
（何を追加・改善するか）

## 背景・理由
（なぜ必要か。ユーザーの課題や技術的な理由を書く）

## 実装方針
- [ ] 〇〇を追加する
- [ ] 〇〇を修正する
- [ ] テストを書く

## 完了条件
（どうなれば完了とみなすか）
```

---

## ラベルのルール

以下のラベルを必ず 1 つ以上付ける。

| ラベル | 用途 |
|--------|------|
| `bug` | バグ・不具合 |
| `enhancement` | 機能追加・改善 |
| `refactor` | リファクタリング |
| `docs` | ドキュメント |
| `blocked` | 他 Issue 待ちで着手できない状態 |

---

## やってはいけないこと

- タイトルを「バグ修正」「対応」など曖昧にしない
- 再現手順のないバグ報告を作らない
- 1 つの Issue に複数の独立した変更を詰め込まない
