# CLAUDE.md（System Development Sample）

このファイルは、システム開発プロジェクトで Claude Code を安定運用するためのサンプルです。  
`CLAUDE.md` としてプロジェクトルートに配置して利用してください。

## Claude Code への運用ルール

- 変更は必ず `探索 -> 計画 -> 実装 -> 検証` の順で進める
- 1セッション1タスクを原則とし、無関係な作業を混在させない
- 作業着手前に必ずチケットを作成し、チケットに紐づく作業のみ実施する
- すべての変更は作業用ブランチで行い、`main` へ直接コミットしない
- マージは必ず Pull Request 経由とし、レビュー承認後に実施する
- 失敗を同じ方法で2回繰り返したら、方針変更か `/clear` を提案する
- 実装完了時は、`差分` と `テスト結果` をセットで報告する
- 不明点は推測で埋めず、前提を明示して最小リスク案で進める

## プロジェクト概要

- プロダクト: 受発注・在庫・請求を管理する業務システム
- 主要利用者: 営業、倉庫、経理、管理者
- 目標: 既存業務の属人化解消、監査可能な操作履歴、障害時復旧の迅速化

## 技術スタック（例）

- Frontend: React + TypeScript
- Backend: Node.js (NestJS) / REST API
- DB: PostgreSQL
- Cache/Queue: Redis
- Infra: AWS (ECS, RDS, S3, CloudWatch)
- CI/CD: GitHub Actions

## 開発セットアップ / 検証コマンド

```bash
# セットアップ
pnpm install
docker compose up -d

# 開発起動
pnpm dev

# 品質チェック
pnpm lint
pnpm typecheck
pnpm test

# E2E（必要時）
pnpm test:e2e
```

## 受け入れ基準（Definition of Done）

- 仕様の受け入れ条件を満たしている
- 変更対象に対応するテストを追加/更新している
- `lint` / `typecheck` / `test` が通過している
- 影響範囲（API、DB、バッチ、権限）を報告している
- 差分に意図しない変更がない

## 実装依頼フォーマット（必須）

依頼時は次の形式を使うこと。

```text
対象: [ファイル/関数/画面/API]
症状: [何が起きているか、再現手順]
制約: [使える技術、禁止事項、互換性要件]
成功条件: [テスト観点、受け入れ条件]
検証コマンド: [実行コマンド]
```

## チケット駆動ルール（必須）

- すべての開発作業はチケット起点で開始する
- チケットには「背景」「目的」「対応範囲」「受け入れ条件」を記載する
- コミットメッセージと PR 本文にチケット番号を明記する
- チケットに記載のないスコープ変更は、追記または別チケット化してから着手する

## Git ブランチ / PR ルール（必須）

- 作業開始時に `main` 最新化後、作業ブランチを作成する
- ブランチ命名規則は `feature/<ticket-id>-<summary>` を基本とする
- `main` / `master` への直接 push は禁止
- 変更完了後は PR を作成し、レビューを依頼する
- 最低1名の承認を得るまでマージしない
- CI（lint / test / typecheck）が失敗している PR はマージしない
- マージ方式はチーム規約（例: squash merge）に統一する

## アーキテクチャと変更ポリシー

- Controller は薄く保ち、業務ロジックは Service 層へ集約する
- DB 変更は「マイグレーション + ロールバック手順」を必須とする
- 外部I/O（メール、決済、S3）はインターフェース経由で抽象化する
- 例外は握り潰さず、ログとエラーコードを定義する

## コーディング規約（抜粋）

- TypeScript は `strict` 前提、`any` は原則禁止
- 公開関数は入力・出力型を明示する
- 命名は業務用語に合わせる（略語の乱用禁止）
- コメントは「なぜ」を補足し、「何を」はコードで表現する

## テスト戦略

- Unit: ドメインロジック、バリデーション、分岐網羅
- Integration: API と DB の整合性、トランザクション境界
- E2E: 受注登録、在庫引当、請求確定などの主要業務フロー
- 回帰防止: バグ修正時は必ず再現テストを追加する

## セキュリティ / コンプライアンス

- 個人情報・機微情報はログに出力しない
- 認可チェックは「画面側」ではなく「API側」で強制する
- SQL インジェクション対策としてプリペアドステートメントを使用する
- 秘密情報は `.env` に置き、リポジトリへコミットしない

## 運用ルール（コンテキスト管理）

- タスク切替時は `/clear` を使い、文脈汚染を防ぐ
- 長時間作業は `/compact` で中間要約を残す
- 調査依頼はスコープを限定する（対象ディレクトリ・機能を明記）
- CLAUDE.md は短く維持し、可変情報は別ドキュメントへ分離する

## 権限 / 危険操作ポリシー

- 原則最小権限で実行する
- 破壊的操作（`DROP`, `TRUNCATE`, `rm -rf`, `git reset --hard`）は禁止
- 本番データに影響する操作は必ず事前承認を取る
- マイグレーション実行前にバックアップとロールバック手順を確認する

## PR / レビュー方針

- PR には「関連チケット・背景・変更内容・テスト結果・影響範囲・ロールバック手順」を記載
- レビューは「バグ/回帰リスク/テスト不足」を優先して指摘する
- 大規模変更は分割PRを原則とする

## 30秒セルフチェック（完了時）

- `git diff` に意図しない差分がない
- テスト/型/lint の結果が揃っている
- 監視・ログ・権限への影響を確認した
- 報告に「再現方法」と「確認結果」を含めた

## 禁止事項

- 検証未実施で「完了」と報告すること
- 根拠なしで設計方針を変更すること
- 1つのPRに無関係な変更を混ぜること
- チケット未作成のまま実装を開始すること
- レビュー承認なしで PR をマージすること
- 本番環境へ直接手作業で修正を入れること
