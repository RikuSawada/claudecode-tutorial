# ベストプラクティス

---

## はじめに

Claude Code を効果的に使うためのノウハウをまとめています。
初心者がやりがちなミスと、経験者が実践しているコツを紹介します。

---

## プロンプトのベストプラクティス

### 1. 役割と文脈を最初に伝える

```
# 良い例
「このプロジェクトは Next.js + TypeScript で作られた EC サイトです。
 商品一覧ページに検索フィルター機能を追加してください。
 フィルターはカテゴリ・価格帯・評価で絞り込めるようにしてください。」

# 悪い例
「フィルター追加して」
```

---

### 2. 制約条件を明示する

```
「以下の制約で実装してください：
- 既存のAPIの仕様は変えない
- TypeScript の型定義を必ず書く
- テストも一緒に書く
- コメントは日本語で書く」
```

---

### 3. 出力形式を指定する

```
# コードを生成させる
「〇〇の実装をしてください」

# 説明を求める
「〇〇の仕組みを箇条書きで説明してください」

# レビューを求める
「このコードの問題点を指摘してください。
 修正はせず、指摘だけしてください」

# 調査を求める
「このエラーの原因を調べて、修正前に原因を説明してください」
```

---

### 4. 段階的に確認しながら進める

大きなタスクは一度に頼まず、段階ごとに確認します：

```
Step 1:「まず現在のコードを読んで、どう実装するか方針を教えて」
Step 2:「その方針で実装して」
Step 3:「テストを書いて」
Step 4:「コミットして」
```

---

### 5. 「なぜ」も伝える

目的を伝えると、Claude がより適切な判断をできます：

```
# 良い例
「レスポンス時間を改善したいので、
 このAPIエンドポイントのクエリをチューニングして」

# 悪い例
「クエリを修正して」
```

## 作業効率を上げるコツ

### CLAUDE.md を充実させる

CLAUDE.md に情報を書けば書くほど、毎回説明する手間が省けます：

```markdown
## よく使うパターン
- 新規APIエンドポイントは controllers/ にハンドラを追加し、routes/ にルートを登録する
- バリデーションは zod を使う
- エラーレスポンスは { error: string, code: string } の形式で返す
```

---

### `/clear` を適切なタイミングで使う

```
# こんなときは /clear でリセット
- 別のタスクに切り替えるとき
- 長い会話で Claude が混乱してきたとき
- 前の文脈が邪魔になってきたとき
```

---

### エラーは全文貼る

エラーの一部だけ貼ると原因特定が難しくなります：

```
# 良い例（エラー全文を貼る）
「以下のエラーを修正して：

Error: Cannot find module './utils/auth'
    at Function.Module._resolveFilename (node:internal/modules/cjs/loader:1039:15)
    at Function.Module._load (node:internal/modules/cjs/loader:885:27)
    at Module.require (node:internal/modules/cjs/loader:1105:19)
    at require (node:internal/modules/cjs/helpers:103:18)
    at Object.<anonymous> (/src/controllers/userController.ts:3:1)」

# 悪い例
「モジュールが見つからないエラーが出る」
```

---

### ファイル名・行番号を指定する

```
「src/controllers/userController.ts の45行目付近のエラー処理を修正して」
```

## コードの品質を保つコツ

### 実装前にレビューを依頼する

```
「実装する前に、どういうアプローチで進めるか教えて。
 問題があればフィードバックする」
```

---

### テストを一緒に書かせる

```
「〇〇機能を実装して、ユニットテストも一緒に書いて」
```

テストを後から書くより、実装と同時に書く方が品質が上がります。

---

### コミット前に差分を確認する

```
「コミットする前に、変更内容の差分を見せて」
```

---

### 既存コードのスタイルに合わせる

```
「このプロジェクトの既存コードのスタイルに合わせて実装して。
 まず既存のコードをいくつか読んでからにして」
```

## チームで使うときのコツ

### CLAUDE.md をチームで管理する

```markdown
# CLAUDE.md
## チームルール
- PRのタイトルは「[Fix]」「[Add]」「[Update]」で始める
- コミットメッセージは日本語で書く
- レビュアーは必ず @team-member を指定する
- ステージング環境へのデプロイは自動（main マージ後）
```

---

### カスタムスキルをチームで共有する

`.claude/commands/` ディレクトリを Git で管理すれば、
チーム全員が同じスキルを使えます：

```
.claude/
└── commands/
    ├── check-security.md    # セキュリティレビュー
    ├── write-tests.md       # テスト生成
    ├── pr-summary.md        # PR説明文生成
    └── review-code.md       # コードレビュー
```

## よくある失敗パターン

### ❌ 一度に大きすぎるタスクを頼む

```
# NG
「ECサイト全体のバックエンドを作って」

# OK：小さく分割する
「まず商品モデルとDBスキーマを作って」
```

---

### ❌ 変更を確認せずに承認する

ファイル変更を承認する前に、必ず内容を確認してください。
特に：
- 予期しないファイルが変更されていないか
- 削除されているものはないか
- 意図と違う実装になっていないか

---

### ❌ 機密情報をそのまま貼る

```
# NG：APIキーをそのまま貼る
「このコードを修正して：
 const API_KEY = 'sk-proj-xxxxxxxxxx'」

# OK：マスクして貼る
「このコードを修正して：
 const API_KEY = process.env.API_KEY」
```

---

### ❌ エラーが出ても同じ依頼を繰り返す

同じ依頼が2回失敗したら、アプローチを変えましょう：

```
「さっきの方法ではうまくいかなかった。
 別のアプローチで試してみて。
 例えば〇〇という方法はどうか?」
```

## 生産性を高める活用例

### 朝のルーティン

```
「昨日から今日にかけての変更をまとめて」
「未解決の Issue を優先度順に並べて」
「今日対応すべき PR のレビューコメントを確認して」
```

### コードを書く前に

```
「この機能を実装する前に、
 関連する既存コードを調べて、影響範囲を教えて」
```

### PR作成時

```
「この変更でPRを作って。
 変更内容・テスト方法・注意事項を含む説明文を書いて」
```

### 週次レポート

```
「今週のコミット履歴から週次レポートを作成して。
 何を実装して、何を修正したかをまとめて」
```
