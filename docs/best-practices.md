# ベストプラクティス

最終更新: 2026-03-04  
公式参照: https://code.claude.com/docs/ja/best-practices

## 1. 最優先原則

- 先に検証条件を与える（テスト・期待値・実行コマンド）
- 変更は「探索→計画→実装→検証」で進める
- 1タスク1セッションを基本にし、無関係タスクを混ぜない
- 失敗を2回繰り返したら `/clear` または新規セッションへ切替
- 生成結果は必ず `diff` とテストで確認する

## 2. 実装依頼テンプレート（推奨）

以下を満たす依頼は、修正往復とトークン消費を減らしやすいです。

- 対象: ファイル/関数/行番号
- 症状: 何が壊れているか、どう再現するか
- 制約: 使用技術、禁止事項、互換性要件
- 成功条件: テストケース、性能目標、受け入れ条件
- 検証: 実行するコマンドと報告形式

例:

```text
src/auth.ts の login() を修正。
空文字パスワード時は ValidationError を返す。
Vitest で正常系3件・異常系2件を追加し、npm test の結果を添えて報告。
```

## 3. 30秒検証チェック

タスク完了時に最低限これを確認します。

- `git diff` に意図しない変更がない
- テスト/型チェック/lint の結果が妥当
- コストが想定内（必要なら `/cost` で確認）

## 4. コンテキスト管理

### 基本

- タスク切替時に `/clear`
- 長時間セッションでは `/compact <instructions>` を計画的に使用
- スコープ外探索を避ける（「全体を調べて」は禁止）

### コスト最適化

- `.claudeignore` で不要ファイルを除外（`node_modules/`, `dist/`, ログなど）
- 大きなファイル全貼りを避け、該当箇所だけ渡す
- 長い会話を続けすぎず、区切って再開する

### 劣化サイン

次が出たらセッションを切り替える。

- 以前の指示を繰り返し忘れる
- 命名や実装方針の一貫性が崩れる
- 関係ないファイルを編集し始める

## 5. 環境整備

### CLAUDE.md

`/init` を起点に、短く保ちます。

- 含める: 推測できないルール、テスト手順、必須コマンド、運用規約
- 含めない: コードを読めば分かる説明、長いチュートリアル、頻繁に変わる情報

### 権限・サンドボックス

- `/permissions` と `/sandbox` を先に設計
- 許可は最小権限を基本にする

### ツール拡張

- CLI連携（`gh`, `aws`, `gcloud` など）
- MCP サーバー
- Hooks（`SessionStart` など）
- Skills / Subagents

## 6. ワークフロー設計

### 5分ルール

- 手作業5分未満: 直接実装
- 手作業5分以上: 先に計画を作らせる

### Subagent分離

- 探索担当と実装担当を分離し、メイン文脈を汚さない
- レビューは別コンテキスト（Writer/Reviewer）で実施

### セッション継続

- `claude --continue`（直近再開）
- `claude --resume`（履歴選択）
- `/rename` で目的別に識別

## 7. 自動化

### ヘッドレス実行

- `claude -p "..."` で非対話運用
- `--output-format json` でCIやスクリプト連携

### ファンアウト

- 複数対象への同種処理は並列化
- `--allowedTools` 等で権限を絞って安全に実行

### 反復作業の定型化

- `.claude/commands/` にスラッシュコマンドを作る
- `.claude/skills/` で手順と知識を再利用する

## 8. よくある失敗

- 1セッションに複数タスクを詰め込む
- 検証なしで「動いた」を採用する
- 同一アプローチで失敗を繰り返す
- CLAUDE.md を肥大化させる
- 調査スコープを指定せずに探索させる

## 9. 導入順（推奨）

1. 具体的な依頼 + 検証条件の明示
2. 30秒検証チェックの定着
3. 1タスク1セッション運用
4. CLAUDE.md 整備
5. Subagent / Hooks / Skills で自動化
