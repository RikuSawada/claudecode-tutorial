# 課題・TODO の管理方法

Claude Code と組み合わせた課題・TODO の管理パターンを紹介します。
プロジェクトの規模やチーム構成に合わせて選んでください。

---

## パターン比較

| パターン | 向いているケース | Claude Code との連携 |
|---------|---------------|-------------------|
| [ローカル Issue ファイル](#パターン1-ローカル-issue-ファイル管理) | 個人・小規模・Git だけで完結したい | ◎ 最もシームレス |
| [GitHub Issues](#パターン2-github-issues) | チーム開発・OSS・PR と紐付けたい | ◎ gh コマンドで直接操作 |
| [Notion](#パターン3-notion-mcp-連携) | チームの情報を一元管理したい | ○ MCP 経由で操作可能 |
| [CLAUDE.md に書く](#パターン4-claudemd-に-todo-を書く) | セッション内の作業メモ程度 | ◎ そのまま読み込まれる |

---

## パターン1: ローカル Issue ファイル管理

プロジェクト内に `issues/` フォルダを作り、Markdown ファイルでチケットを管理します。
外部サービス不要で Git だけで完結し、Claude Code が直接ファイルを読んで作業できます。

### フォルダ構成

```
your-project/
└── issues/
    ├── open/
    │   ├── 001-ログイン機能の実装.md
    │   ├── 002-パスワードリセット機能.md
    │   └── 003-タスク一覧のソート機能.md
    ├── in-progress/
    │   └── 004-通知メール送信機能.md
    └── closed/
        └── 000-初期セットアップ.md
```

### チケットのテンプレート

```markdown
# [タイトル]

## 概要

<!-- この Issue で何をするか1〜2文で -->

## 背景・目的

<!-- なぜこの作業が必要か -->

## タスク

- [ ] タスク1
- [ ] タスク2
- [ ] タスク3

## 受け入れ条件

- [ ] 〇〇ができること
- [ ] 〇〇エラーが出ないこと

## 参考

<!-- 関連ファイル・参考リンクなど -->
```

### チケットファイルの例

`issues/open/002-パスワードリセット機能.md`:

```markdown
# パスワードリセット機能

## 概要

メールアドレスを入力するとリセット用のリンクをメールで送信し、
リンクから新しいパスワードを設定できるようにする。

## 背景・目的

ログインできなくなったユーザーのサポート問い合わせが増えているため。

## タスク

- [ ] リセット用トークンの生成と保存（有効期限: 1時間）
- [ ] メール送信処理の実装
- [ ] パスワード再設定フォームの作成
- [ ] トークン検証と期限切れチェック

## 受け入れ条件

- [ ] 存在しないメールアドレスを入力した場合もエラーを返さない（セキュリティ）
- [ ] リセットリンクは1回使用したら無効になる
- [ ] 期限切れリンクにアクセスした場合、わかりやすいエラーページを表示する

## 参考

- 関連ファイル: `src/routers/auth.py`, `src/services/email.py`
- 仕様書: `docs/specs/auth.md`
```

### Claude Code での活用

```
「issues/open/ にあるチケットを一覧表示して、優先度順に並べて」

「issues/open/002-パスワードリセット機能.md を読んで、実装を開始してください」

「実装が完了しました。issues/in-progress/002-パスワードリセット機能.md を
 issues/closed/ に移動して、完了日を記載してください」
```

```
「新しいチケットを作成してください：
 タイトル: タスクの検索機能
 概要: タイトル・優先度・期限でタスクを絞り込める検索機能を追加する
 ファイルは issues/open/ に保存してください」
```

### ステータス管理の運用

```
open/        → 未着手のチケット
in-progress/ → 現在対応中（基本1〜2枚まで）
closed/      → 完了したチケット
```

完了チケットにはヘッダーに日付を追記する運用が便利です：

```markdown
# パスワードリセット機能
**完了日**: 2026-03-05 | **PR**: #42
```

## パターン2: GitHub Issues

GitHub Issues を使う方法です。チームでの開発・OSS・PR と紐付けた管理に最適です。
Claude Code は `gh` コマンドを通じて直接操作できます。

### 事前準備

```bash
brew install gh
gh auth login
```

### Claude Code での活用

#### Issue を確認・整理する

```
「未解決の Issue 一覧を表示して、優先度の高いものから並べて」

「Issue #15 の内容を読んで、実装方針を説明してください」

「自分にアサインされている Issue のうち、
 今週中に対応すべきものをリストアップして」
```

#### Issue を作成する

```
「以下の内容で GitHub Issue を作成してください：
 タイトル: タスクの検索機能を追加
 本文: タイトル・優先度・期限でタスクを絞り込める検索機能
 ラベル: enhancement
 アサイン: @me」
```

#### Issue を元に実装する

```
「Issue #20 を読んで、ブランチを作成してから実装してください。
 完了したら PR を作成して Issue と紐付けてください」
```

### Issue テンプレートを設定する

`.github/ISSUE_TEMPLATE/feature.md` を作成しておくと、
Claude Code が Issue を作成する際の品質が安定します：

```markdown
---
name: 機能追加
about: 新しい機能の追加を提案する
---

## 概要

<!-- この機能で何を実現するか -->

## 背景・目的

<!-- なぜこの機能が必要か -->

## 受け入れ条件

- [ ] 〇〇ができること
- [ ] 〇〇エラーが出ないこと

## 補足

<!-- 参考リンク・関連 Issue など -->
```

### ラベルで状態管理する

| ラベル | 意味 |
|-------|------|
| `status: open` | 未着手 |
| `status: in-progress` | 対応中 |
| `status: review` | レビュー待ち |
| `priority: high` | 優先度高 |
| `bug` | バグ |
| `enhancement` | 機能追加 |

```
「Issue #20 に 'status: in-progress' ラベルを付けてください」
```

## パターン3: Notion MCP 連携

Notion を課題管理のデータベースとして使い、MCP 経由で Claude Code から操作します。
ドキュメント・課題・情報を一元管理したいチームに向いています。

### MCP の設定

`~/.claude/settings.json` または `.claude/settings.json`：

```json
{
  "mcpServers": {
    "notion": {
      "command": "npx",
      "args": ["-y", "@notionhq/notion-mcp-server"],
      "env": {
        "OPENAPI_MCP_HEADERS": "{\"Authorization\": \"Bearer your-integration-token\", \"Notion-Version\": \"2022-06-28\"}"
      }
    }
  }
}
```

Notion インテグレーショントークンの取得：
1. [Notion Developers](https://www.notion.so/my-integrations) でインテグレーションを作成
2. 生成されたトークンを上記の `your-integration-token` に設定
3. 対象の Notion ページにインテグレーションを接続

### Notion データベースの構成例

課題管理用のデータベースに以下のプロパティを持たせます：

| プロパティ | 種類 | 説明 |
|----------|-----|------|
| タイトル | タイトル | チケット名 |
| ステータス | セレクト | 未着手 / 進行中 / 完了 |
| 優先度 | セレクト | 高 / 中 / 低 |
| 担当者 | ユーザー | 担当メンバー |
| 期限 | 日付 | 対応期限 |
| 関連 PR | URL | GitHub PR のリンク |

### Claude Code での活用

```
「Notion の課題データベースから、ステータスが『未着手』の
 チケットを優先度順に一覧表示して」

「Notion に新しい課題チケットを作成してください：
 タイトル: タスクの検索機能
 優先度: 高
 期限: 2026-03-31」

「Issue #20 の実装が完了しました。
 Notion の対応するチケットのステータスを『完了』に更新して、
 関連 PR に #42 を記載してください」
```

## パターン4: CLAUDE.md に TODO を書く

最もシンプルな方法です。セッション内で Claude Code に直接読ませながら進める作業に向いています。
外部ツールを使わず、現在の作業タスクをその場で管理したい場合に使います。

### CLAUDE.md に TODO セクションを追加

```markdown
## 現在のタスク

### 今週の対応項目
- [ ] パスワードリセット機能の実装 (優先度: 高)
- [ ] タスク一覧のソート機能 (優先度: 中)
- [ ] 通知メールのテンプレート修正 (優先度: 低)

### 完了済み
- [x] ログイン機能のバグ修正 (2026-03-03)
- [x] ユーザー登録フォームのバリデーション追加 (2026-03-02)
```

### Claude Code での活用

```
「CLAUDE.md の今週の対応項目を読んで、
 優先度の高い順に今日やるべきタスクを整理して」

「パスワードリセット機能の実装が完了しました。
 CLAUDE.md の TODO を更新してください」
```

### 注意点

- タスクが増えてきたら [パターン1](#パターン1-ローカル-issue-ファイル管理) や [パターン2](#パターン2-github-issues) に移行する
- CLAUDE.md は 200 行以内に収めるのが推奨（それ以降は切り捨てられる）
- 個人の作業メモ程度の用途に留めるのが適切

## パターンの選び方

```
個人プロジェクト・小規模
├── Git だけで完結したい  → パターン1（ローカル Issue ファイル）
└── シンプルに作業メモ   → パターン4（CLAUDE.md）

チーム開発
├── GitHub を中心に管理  → パターン2（GitHub Issues）
└── Notion で情報一元化  → パターン3（Notion MCP）
```

複数を組み合わせることも可能です。例えば：

- **日常の作業管理** → GitHub Issues（PR と紐付けて管理）
- **大きな機能の詳細仕様** → Notion（ドキュメントとして管理）
- **現在の作業状況** → CLAUDE.md に TODO メモ
