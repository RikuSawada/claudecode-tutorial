# 開発ワークフローでの活用

Claude Code を要件定義からリリースまでの各フェーズで活用する方法をまとめています。

---

## 全体フロー

```
要件定義 → 設計 → 製造 → UT → IT → ST → リリース
```

各フェーズで Claude Code が担う役割と、具体的なプロンプト例を紹介します。

---

## 1. 要件定義

### Claude Code でできること

- 顧客・ステークホルダーの要望をユーザーストーリーに整理する
- 受け入れ条件（Acceptance Criteria）を定義する
- 要件の曖昧さや矛盾を指摘する
- 類似サービスの機能を調査して要件の抜け漏れを検出する

### プロンプト例

```
「以下の要望をユーザーストーリー形式に整理してください：

・ユーザーがタスクを登録できるようにしたい
・タスクに期限を設定したい
・期限が近づいたら通知がほしい

形式: As a [ユーザー], I want to [機能], so that [目的].」
```

```
「以下の要件定義書を読んで、
 曖昧な点・矛盾している点・抜け漏れていそうな点を指摘してください：

[要件定義書の内容]」
```

```
「タスク管理アプリの要件として、
 競合サービス（Todoist・Asana・Notion）が持っている
 標準的な機能の一覧を作成してください」
```

### 成果物例

```markdown
## ユーザーストーリー

### タスク登録
- As a ユーザー, I want to タスクにタイトル・期限・優先度を設定できる,
  so that 重要なタスクを見落とさずに管理できる

### 通知
- As a ユーザー, I want to 期限の24時間前にメール通知を受け取れる,
  so that 期限切れを防げる

## 受け入れ条件（タスク登録）
- [ ] タイトルは必須（最大100文字）
- [ ] 期限は任意。過去の日付は設定不可
- [ ] 優先度は「高・中・低」から選択
```

## 2. 設計

### Claude Code でできること

- データベース設計（ER 図・テーブル定義）を作成する
- API 設計（OpenAPI/Swagger 形式）を作成する
- ディレクトリ構成・アーキテクチャを提案する
- 設計書のレビューと問題点の指摘

### プロンプト例

```
「以下のユーザーストーリーを元に、
 PostgreSQL 用のテーブル定義を作成してください。
 カラム名・型・制約・インデックスを含めてください：

[ユーザーストーリーの内容]」
```

```
「タスク管理アプリの REST API を設計してください。
 OpenAPI 3.0 形式で、以下のエンドポイントを含めてください：
 - タスクの CRUD
 - ユーザー認証（ログイン・ログアウト）」
```

```
「以下の設計書を読んで、
 パフォーマンス・セキュリティ・保守性の観点で問題点を指摘してください：

[設計書の内容]」
```

### 成果物例

```sql
-- テーブル定義
CREATE TABLE tasks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id),
    title       VARCHAR(100) NOT NULL,
    due_date    DATE,
    priority    VARCHAR(10) CHECK (priority IN ('high', 'medium', 'low')),
    status      VARCHAR(20) NOT NULL DEFAULT 'pending',
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_tasks_user_id ON tasks(user_id);
CREATE INDEX idx_tasks_due_date ON tasks(due_date);
```

## 3. 製造（実装）

### Claude Code でできること

- 設計書に基づいてコードを実装する
- 実装済みコードのレビューと改善提案
- リファクタリング
- コードの説明・ドキュメント生成

### プロンプト例

```
「以下のテーブル定義を元に、FastAPI のエンドポイントと
 SQLAlchemy モデルを実装してください。
 プロジェクトの既存コードスタイルに合わせてください：

[テーブル定義]」
```

```
「src/routers/tasks.py を読んで、以下の観点でレビューしてください：
 - バグの可能性
 - セキュリティの問題
 - パフォーマンスの問題
 - コードの可読性」
```

```
「この関数を読みやすくリファクタリングしてください。
 処理の意味は変えず、変数名と構造を改善してください：

[対象コード]」
```

### 実装の進め方（推奨）

大きな機能は以下のステップで進めると安全です：

```
Step 1: 「〇〇機能をどう実装するか方針を説明して」
        → Claude の方針を確認・フィードバック

Step 2: 「その方針でモデル・スキーマを実装して」
        → レビューして承認

Step 3: 「次にエンドポイントを実装して」
        → レビューして承認

Step 4: 「実装が完了したのでコミットして」
```

## 4. UT（単体テスト）

### Claude Code でできること

- 実装したコードのユニットテストを生成する
- テストケースの正常系・異常系・エッジケースを網羅する
- モック・フィクスチャを作成する
- テストカバレッジのレポートを確認して不足を補う

### プロンプト例

```
「src/crud/tasks.py の各関数に対して pytest のユニットテストを書いてください。
 - 正常系・異常系・エッジケースを網羅する
 - 既存の conftest.py のフィクスチャを活用する
 - モックが必要な箇所は unittest.mock を使う」
```

```
「テストを実行して、カバレッジが低い箇所を特定してください。
 その箇所に追加のテストケースを書いてください」
```

```
「以下のテストが失敗しています。原因を特定して修正してください：

FAILED tests/test_tasks.py::test_create_task - AssertionError: 422 != 201」
```

### テスト実行

```bash
# カバレッジ付きでテスト実行
pytest --cov=app --cov-report=term-missing

# 特定ファイルのみ
pytest tests/test_tasks.py -v
```

## 5. IT（結合テスト）

### Claude Code でできること

- API 間の結合テストシナリオを作成する
- テストデータ・シードデータを準備する
- 認証フローを含むエンドツーエンドのシナリオを実装する
- 外部サービス（メール・決済など）のモックを作成する

### プロンプト例

```
「タスク管理アプリの結合テストシナリオを作成してください。
 以下のフローをカバーしてください：
 1. ユーザー登録 → ログイン
 2. タスク作成 → 一覧取得 → 更新 → 削除
 3. 認証エラー（未ログイン・権限なし）のケース」
```

```
「IT 用のテストデータ（SQL の INSERT 文）を作成してください。
 ユーザー 3 名・各ユーザーにタスク 5 件の構成で」
```

```
「外部メール送信サービス（SendGrid）をモックして、
 通知機能の結合テストを実装してください」
```

### テストシナリオ例

```python
# tests/integration/test_task_flow.py

def test_full_task_lifecycle(client, auth_headers):
    """タスクの作成から削除までの一連のフローをテスト"""

    # タスク作成
    res = client.post("/api/tasks", json={
        "title": "IT テスト用タスク",
        "due_date": "2026-03-31",
        "priority": "high"
    }, headers=auth_headers)
    assert res.status_code == 201
    task_id = res.json()["id"]

    # 一覧に存在することを確認
    res = client.get("/api/tasks", headers=auth_headers)
    assert any(t["id"] == task_id for t in res.json())

    # 更新
    res = client.patch(f"/api/tasks/{task_id}", json={"priority": "low"}, headers=auth_headers)
    assert res.status_code == 200

    # 削除
    res = client.delete(f"/api/tasks/{task_id}", headers=auth_headers)
    assert res.status_code == 204
```

## 6. ST（システムテスト）

### Claude Code でできること

- ST のテストケース一覧を作成する
- バグ報告書のテンプレートを生成する
- 発見されたバグの原因調査と修正
- リグレッションテストの自動化スクリプトを作成する

### プロンプト例

```
「タスク管理アプリの ST テストケース一覧を作成してください。
 以下の観点を含めてください：
 - 機能テスト（全ユーザーストーリーの検証）
 - 異常系（不正入力・権限エラー）
 - 性能テスト（同時接続 100 ユーザー）
 - セキュリティテスト（SQLインジェクション・XSS）」
```

```
「以下のバグ報告を元に、原因を調査して修正してください：

【バグ内容】
期限を設定せずにタスクを作成すると 500 エラーが発生する

【再現手順】
1. ログインする
2. タスク作成フォームで期限を空欄のまま送信する
3. 500 Internal Server Error が返る

【期待動作】
期限なしでタスクが作成される（201 が返る）」
```

```
「ST で発見されたバグの一覧を元に、
 リグレッションテストを pytest で実装してください：

[バグ一覧]」
```

### テストケース表の例

```markdown
| # | テスト項目 | 操作手順 | 期待結果 | 結果 |
|---|-----------|---------|---------|------|
| 1 | タスク正常作成 | タイトル・期限・優先度を入力して送信 | 201 が返り一覧に表示 | ○ |
| 2 | タイトル未入力 | タイトル空欄で送信 | バリデーションエラー 422 | ○ |
| 3 | 過去の期限指定 | 昨日の日付を期限に設定して送信 | バリデーションエラー 422 | - |
| 4 | 未認証でのアクセス | ログインせずにタスク一覧を取得 | 401 が返る | - |
```

## 7. リリース

### Claude Code でできること

- リリースノートを自動生成する
- デプロイ手順書を作成・確認する
- リリース前チェックリストを生成する
- デプロイスクリプトのレビューと問題点の指摘

### プロンプト例

```
「今回のリリース（v1.2.0）のコミット履歴を読んで、
 ユーザー向けのリリースノートを作成してください。
 技術的な内容は平易な言葉に言い換えてください」
```

```
「以下のデプロイスクリプトをレビューして、
 問題点・抜け漏れ・改善点を指摘してください：

[deploy.sh の内容]」
```

```
「本番リリース前のチェックリストを作成してください。
 以下の観点を含めてください：
 - 環境変数・設定ファイルの確認
 - DB マイグレーションの確認
 - バックアップの確認
 - ロールバック手順の確認」
```

### リリースノート例

```markdown
## v1.2.0 リリースノート（2026-03-03）

### 新機能
- タスクに優先度（高・中・低）を設定できるようになりました
- 期限が近いタスクをメールでお知らせする通知機能を追加しました

### 改善
- タスク一覧の表示速度を改善しました（約 40% 高速化）

### バグ修正
- 期限を設定せずにタスクを作成するとエラーになる問題を修正しました

### 注意事項
- DB マイグレーションが必要です。デプロイ前に実行してください
```

### リリース前チェックリスト

```markdown
## リリース前チェックリスト

### コード・テスト
- [ ] すべてのテストが通過している
- [ ] ST の指摘事項がすべて解消されている
- [ ] コードレビューが完了している

### インフラ・環境
- [ ] 本番環境の環境変数が最新になっている
- [ ] DB マイグレーションスクリプトを確認した
- [ ] バックアップが取得済みである

### ロールバック準備
- [ ] ロールバック手順を確認した
- [ ] 前バージョンのデプロイイメージが残っている
- [ ] DB のロールバック SQL を用意した

### リリース後
- [ ] 本番環境で動作確認した
- [ ] エラーログに異常がないことを確認した
- [ ] 監視アラートが正常に動作している
```

## フェーズをまたいだ活用パターン

### Issue 駆動の開発フロー

GitHub と連携することで、Issue を起点に設計〜リリースまでを一気通貫で進められます：

```
1. 要件を Issue に登録
   「以下の要件を GitHub Issue として登録してください：[要件内容]」

2. Issue を読んで実装
   「Issue #12 を読んで、設計方針を説明してから実装してください」

3. テストを書いてコミット
   「実装が完了したのでテストを書いて、コミットしてください」

4. PR を作成
   「この変更の PR を作成してください。Issue #12 と紐付けてください」

5. レビュー対応
   「PR #15 のレビューコメントに対応してください」
```

### CLAUDE.md でフェーズ情報を共有する

プロジェクトの現在のフェーズや制約を CLAUDE.md に記載しておくと、
毎回説明しなくて済みます：

```markdown
## 現在のフェーズ

ST 中。新規実装は行わず、バグ修正のみ行うこと。
テスト修正は必ず対応するリグレッションテストを追加してからコミットすること。
```
