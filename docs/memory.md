# Memory（記憶機能）

---

## Memory とは

Claude Code の Memory は、**セッションをまたいで情報を記憶させる仕組み** です。

通常、Claude Code はセッションを終了すると会話内容を忘れます。
Memory を活用することで：

- プロジェクトの文脈を毎回説明しなくて済む
- 好みの作業スタイルを覚えさせられる
- よく使う情報をいつでも参照できる

---

## Memory の種類

| 種類 | 場所 | 用途 |
|-----|------|------|
| **プロジェクトメモリ** | `.claude/CLAUDE.md` / `CLAUDE.md` | プロジェクト固有の情報 |
| **グローバルメモリ** | `~/.claude/CLAUDE.md` | 全プロジェクト共通の設定・好み |
| **自動メモリ** | `~/.claude/projects/*/memory/` | Claude が自動で保存する記憶 |

---

## グローバルメモリの活用

`~/.claude/CLAUDE.md` に書いた内容は、**すべてのプロジェクトで有効** になります。

### 設定例

```markdown
# グローバル設定

## 作業スタイル
- コードの説明は日本語でお願いします
- コミットメッセージは日本語で書いてください
- 実装前に必ず方針を説明してから作業してください
- コードを変更する前に既存のコードを読んでください

## 技術的な好み
- TypeScript を優先して使う
- テストは Vitest を使う
- スタイルは Tailwind CSS を使う
- パッケージマネージャーは npm を使う

## 禁止事項
- `console.log` をデバッグ用に残さない
- `any` 型を使わない
- `--no-verify` でコミットフックをスキップしない
```

## 自動メモリ（Auto Memory）

Claude Code は作業を通じて学習した情報を自動で記憶します。

### 保存される場所

```
~/.claude/projects/
└── -Users-yourname-your-project/
    └── memory/
        ├── MEMORY.md      # 常に読み込まれるメインの記憶
        ├── patterns.md    # コードパターン・規約
        └── debugging.md   # デバッグのノウハウ
```

### 何が記憶されるか

- プロジェクトのアーキテクチャや設計方針
- 繰り返し確認された規約・パターン
- 過去に解決したバグや問題の解決策
- ユーザーの好みや作業スタイル

---

## 記憶を明示的に依頼する

Claude に特定の情報を記憶させたいときは、明示的に依頼します：

```
「このプロジェクトでは zod を使ったバリデーションが標準です。
 これを覚えておいてください」

「コミットメッセージは必ず日本語で書いてください。これをずっと守ってください」

「認証処理は src/lib/auth.ts に集約されています。
 認証関連の変更は必ずこのファイルを参照するよう覚えておいてください」
```

---

## 記憶を確認する

```
「今のプロジェクトについて何を覚えていますか？」
「私の作業スタイルの好みを教えてください」
「このプロジェクトのアーキテクチャについて知っていることを説明してください」
```

---

## 記憶を忘れさせる

間違った情報や古い情報を修正したいとき：

```
「さっき伝えた〇〇については忘れてください」
「〇〇についての認識が間違っていました。正しくは〇〇です」
「メモリから〇〇の情報を削除してください」
```

---

## MEMORY.md を直接編集する

自動メモリファイルは通常のテキストファイルなので、直接編集できます：

```bash
# グローバルメモリを開く
code ~/.claude/CLAUDE.md

# プロジェクトのメモリを開く
code ~/.claude/projects/-Users-yourname-your-project/memory/MEMORY.md
```

## 記憶を活用した効率的な開発

### 初回セッション（情報を教える）

```
「このプロジェクトについて説明します：
- Next.js 14 + TypeScript + Prisma + PostgreSQL の構成
- 認証は NextAuth.js を使用
- スタイルは Tailwind CSS
- テストは Vitest + Testing Library
- API は /app/api/ ディレクトリに配置
- DBスキーマは prisma/schema.prisma で管理

これを覚えておいてください」
```

### 2回目以降のセッション

```
「前回の続きをやります。このプロジェクトの構成は把握していますか？」
```

## チームでの Memory 活用

### 共有できる記憶（Git 管理）

`CLAUDE.md`（プロジェクトルート）は Git でコミットして共有できます：

```bash
git add CLAUDE.md
git commit -m "CLAUDE.md にプロジェクト情報を追記"
```

### 個人の記憶（Git 管理しない）

`~/.claude/` 以下の個人設定は Git 管理せず、各自が設定します。

## Memory のベストプラクティス

### こまめに更新する

プロジェクトの変更に合わせて CLAUDE.md を更新しましょう：

```
「今日追加した認証機能の仕組みを CLAUDE.md に追記してください」
「アーキテクチャが変わったので、CLAUDE.md を最新の状態に更新して」
```

### 重要な決定事項を記録する

```
「今日の設計議論で決めたことを CLAUDE.md に記録してください：
- 状態管理は Zustand を使う（Redux は使わない）
- API は REST（GraphQL は使わない）
- 認証トークンは HttpOnly Cookie で管理」
```

### 定期的に整理する

```
「CLAUDE.md の内容が古くなっていないか確認して、
 必要であれば最新の状態に整理してください」
```

---

## 注意事項

- MEMORY.md の先頭200行が自動で読み込まれます（それ以降は切り捨て）
- 情報が多すぎると重要な内容が埋もれます。**簡潔にまとめる** のがポイント
- 機密情報（パスワード・APIキー）はメモリに書かないでください
- 不正確な情報が記憶されると判断ミスの原因になります。定期的に見直しましょう
