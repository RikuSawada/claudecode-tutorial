# プロンプトテンプレート集

AI への依頼は「何を・どの制約で・何を出力させるか」の3点を明示すると精度が上がる。
以下のテンプレートをベースに、プロジェクト固有の情報を埋めて使う。

---

## 要件定義フェーズ

### Issue 下書き生成

```
以下の機能を追加したい。GitHub Issue の下書きを作って。

【機能の概要】
{機能の説明を1〜3行で}

【背景・目的】
{なぜ必要か}

【制約・前提】
{使用できるAPI・既存の仕様との関係・除外事項}

出力形式：
- 背景・目的
- スコープ（やること/やらないこと）
- 受入条件（AC）：箇条書きで具体的に
- 影響範囲（API/DB/画面/運用）
- テスト観点（UT/IT/E2E）
- 仕様の未決事項（Questions）
```

### 要件の曖昧さチェック

```
以下の要件定義を読んで、仕様が曖昧な点・未定義の箇所・実装前に確認が必要な事項をすべて列挙して。
判断や決定はしなくていい。洗い出しだけ。

{Issue または要件文書の内容}
```

### 非機能要件の洗い出し

```
以下の機能要件に対して、見落としやすい非機能要件を列挙して。
- パフォーマンス（レイテンシ・スループット）
- セキュリティ（認証・認可・入力検証）
- 可用性・障害対応
- 運用・監視
- スケーラビリティ

{機能の概要}
```

## 設計フェーズ

### API 設計

```
以下の要件に対して、REST API の設計を提案して。

【要件】
{AC・影響範囲・制約}

【既存の API 構成】
{既存エンドポイントの一覧または参照先ファイルパス}

出力形式：
- エンドポイント一覧（メソッド・パス・概要）
- リクエスト/レスポンスのスキーマ（JSON）
- エラーレスポンスの設計
- 認証・認可の要件
```

### DB スキーマ設計

```
以下の要件に対して、DB スキーマを設計して。

【要件】
{保存するデータ・関係性・制約}

【既存テーブル】
{関連テーブルの定義または参照先}

出力形式：
- テーブル定義（カラム名・型・制約）
- インデックス設計（理由付き）
- 既存テーブルへの影響
- マイグレーション方針
```

### ADR（アーキテクチャ決定記録）の下書き

```
以下の技術選定・設計判断について ADR の下書きを作って。

【決定事項】
{何を決めたか}

【検討した選択肢】
{選択肢A / 選択肢B / ...}

【決定した理由】
{なぜこれを選んだか}

ADR テンプレート：
- タイトル
- 日付
- ステータス（提案中/承認済み/廃止）
- コンテキスト（背景・なぜ決める必要があったか）
- 決定
- 根拠
- 結果（メリット・デメリット・リスク）
```

## 実装フェーズ

### 機能実装依頼（Codex 向け）

```
以下の機能を実装して。

【実装する機能】
{機能の概要}

【参照先】
- 実装パターン：{既存の類似実装のファイルパス}
- 利用するライブラリ：{使うライブラリ名・用途}

【制約】
- 変更禁止ファイル：src/auth/, src/payment/
- エラー処理：{既存の例外処理パターンに従う / 詳細}
- ログ：{ログレベルの方針}

【完了条件】
- UT が通ること
- 既存テストが壊れていないこと
```

### テストコード生成

```
以下の関数・クラスに対して、UT を作成して。

【対象】
{ファイルパス・関数名・クラス名}

【テストフレームワーク】
{Jest / Vitest / pytest など}

【モック方針】
{外部依存はモックする / テスト DB を使う など}

含めること：
- 正常系（主要なパターン）
- 境界条件（空・最大値・最小値・null など）
- 異常系（例外・エラーケース）
- 各テストに「何を検証しているか」のコメント
```

### リファクタリング依頼

```
以下のコードをリファクタリングして。動作は変えないこと。

【対象ファイル】
{ファイルパス}

【リファクタの目的】
{可読性向上 / 重複除去 / 責務分離 など}

【制約】
- 外部インターフェース（API・型定義）は変更しない
- 既存テストが通ること
- パフォーマンスを劣化させない
```

## レビュー・PR フェーズ

### PR 説明文の生成（Claude Code 向け）

```
以下の変更内容をもとに、PR の説明文を作って。

【変更の概要】
git diff main の内容（または「git diff を確認して」）

PR テンプレートに従って出力：
- 目的
- 変更点（箇条書き）
- 動作確認内容
- 追加・更新したテスト
- リスクと対策
- ロールバック手順

レビュー観点も最後に列挙して（人間が特に確認すべき箇所）。
```

### レビュー支援

```
このPRの以下の観点でレビューして：

【確認してほしい観点】
1. {観点1}（例：SQLインジェクションのリスク）
2. {観点2}（例：テストのアサーションが意味のある検証をしているか）
3. {観点3}（例：既存の命名規約に沿っているか）

懸念点は根拠と重要度（high/medium/low）を付けて列挙。
最終判断は自分でするので、決定はしなくていい。

{PR の差分または対象ファイルパス}
```

### セキュリティレビュー（人間必須・AIによる一次確認）

> 最終判断は必ず人間が行う。AI は「懸念点の列挙」に使い、「問題なし」の確定には使わない。

```
以下のコードに対してセキュリティ観点でレビューして。
懸念点と根拠の列挙だけでよい。決定はしなくていい。

【対象ファイル / PR 差分】
{ファイルパスまたは git diff の内容}

以下の観点をすべて確認すること：

1. 認証・認可
   - 権限チェックが漏れている箇所はないか
   - 権限昇格・水平越権が可能な経路はないか
   - 管理者操作に適切なガードがかかっているか

2. 入力バリデーション
   - SQLインジェクション（パラメータのエスケープ・プリペアドステートメント）
   - XSS（出力時のエスケープ・Content-Security-Policy）
   - CSRF（トークン検証・SameSite Cookie）
   - ファイルアップロード（拡張子・サイズ・保存先の検証）

3. 秘密情報の扱い
   - APIキー・トークン・パスワードのハードコードがないか
   - ログに秘密情報が混入する経路はないか
   - レスポンスに不要な内部情報が含まれていないか

4. 依存関係
   - 追加・更新したライブラリに既知の脆弱性はないか（CVE）

出力形式：
- 懸念点（重要度：high / medium / low）
- 懸念の根拠（コードの該当箇所を引用）
- 確認・修正の方針（提案のみ）
```

## デバッグフェーズ

### エラー解析

```
以下のエラーが発生している。原因の仮説を列挙して。

【エラーメッセージ】
{エラーの全文}

【再現手順】
{操作手順を番号で}

【関連ファイル】
{エラーが発生しているファイルパス}

出力：
1. 原因の仮説（可能性が高い順に3〜5個）
2. 各仮説の確認方法
3. 推奨する調査手順
```

### 最小再現の作成

```
以下のバグを最小限のコードで再現できるサンプルを作って。

【バグの概要】
{何が起きているか}

【実行環境】
{OS・ランタイムのバージョン}

【現在のコード】
{問題のあるコードのスニペット}

最小再現ができたら、根本原因の仮説も一緒に出して。
```

### 回帰テストの追加

```
以下のバグを修正した。同じバグが再発しないよう、回帰テストを追加して。

【バグの内容】
{何が起きていたか}

【修正内容】
{どのように直したか}

【テスト対象ファイル】
{修正したファイルパス}

テストには「このテストが何を防いでいるか」のコメントを入れること。
```

## CLAUDE.md 生成

### 初期 CLAUDE.md の生成（新規プロジェクト時）

```
このリポジトリの構成を分析して、CLAUDE.md の叩き台を作って。

以下の項目を含めること：
1. プロジェクト概要（目的・技術スタック・主要コンポーネント）
2. ディレクトリ構成（各ディレクトリの役割）
3. コーディング規約（既存コードから読み取ったもの）
   - 命名規則
   - 例外処理パターン
   - ログの書き方
4. テスト方針（使用フレームワーク・モック方針）
5. AIへの禁止事項（触ってはいけないファイル・領域）
6. よく参照するファイル（エントリポイント・共通ユーティリティ）

推測で書かずに、確認できる情報だけ記載して。不明な点は「要確認」と明記。
```

## プロンプトを資産化する

チームでよく使うプロンプトは以下に保存してバージョン管理する：

```
docs/prompts/
  requirements.md    ← このファイルの「要件定義フェーズ」セクション相当
  design.md          ← 「設計フェーズ」相当
  implementation.md  ← 「実装フェーズ」相当
  review.md          ← 「レビュー・PRフェーズ」相当
  debug.md           ← 「デバッグフェーズ」相当
```

プロンプトの改善は PR で行い、チームで合意してからマージする。
「このプロンプトを使ったら精度が上がった・下がった」を都度フィードバックして育てる。
