# 開発フロー標準

## 全体像

```
要件定義 → 設計 → 実装 → テスト → レビュー → リリース → 運用
   ↑                                              ↓
   └───────────── フィードバック ──────────────────┘

各工程で Claude Code（マネージャー）と Codex（開発者）が役割分担して動く
```

> **Claude Code** が「何を・どう進めるか」を決め、**Codex** が「実際にコードを書く」。
> 逆にすると方針ブレと手戻りが増える。

---

## 役割分担

### Claude Code：マネージャー・アーキテクト

| やること | 具体的な作業 |
|---------|------------|
| 計画・分解 | タスク分割、スコープ整理、未決事項の洗い出し |
| 設計支援 | API/DB/画面設計のレビュー、アーキテクチャの一貫性確認 |
| Issue管理 | Issue下書き生成、受入条件（AC）の精緻化 |
| PR支援 | PR説明文生成、差分要約、レビュー観点の提示 |
| ドキュメント | ADR、設計書、README の生成・更新 |

### Codex：開発者（実装担当）

| やること | 具体的な作業 |
|---------|------------|
| 実装 | 機能実装、リファクタリング |
| テスト | UT/IT のテストコード生成・保守 |
| E2E | Playwright 等を使ったブラウザ操作の自動化 |
| ドキュメント更新 | コード変更に伴うコメント・README 更新 |

### 使い分けのコツ

> 「何を作るか・どう進めるか」 = Claude Code
> 「実際にコードを書く・動かす」 = Codex
> 逆にすると方針ブレと手戻りが増える

---

## 工程別 AI 活用ポイント

### 1. 要件定義

**AI に任せること**
- ユーザーストーリーのテンプレ化と AC（受入条件）作成
- 非機能要件（SLO・セキュリティ・運用）の洗い出し
- 仕様の曖昧さの指摘・ヒアリング項目の列挙

**人間がやること**
- ビジネス判断・優先順位の決定
- ステークホルダーとの合意

**成果物**
- GitHub Issue（背景・AC・スコープ・テスト観点）

---

### 2. 設計

**AI に任せること**
- API 設計（エンドポイント・リクエスト/レスポンス）
- DB スキーマ設計（テーブル定義・インデックス）
- 画面遷移・状態設計
- 例外系・エラーハンドリング設計
- ログ・監視観点の洗い出し
- ADR（アーキテクチャ決定記録）の下書き

**人間がやること**
- アーキテクチャの最終決定
- セキュリティ要件の確認（認証・認可・データ保護）
- 既存システムへの影響評価

**成果物**
- 設計書（`docs/design/`）、ADR（`docs/adr/`）

---

### 3. 実装

**AI に任せること**
- 機能実装（Codex が担当）
- テストコード生成（UT/IT）
- リファクタリング
- ドキュメントコメントの追加・更新

**人間がやること**
- 生成されたコードの確認・承認
- セキュリティ・根幹領域の実装レビュー（必須）

**実装の基本サイクル**

```
1. Claude Code でタスクを分解・Issue を整理
2. ブランチを切る（feature/issue-{番号}-{概要}）
3. Codex に実装させる（小さく）
4. テストを実行して確認
5. 小さくコミット
6. 手順3〜5 を繰り返す
7. PR を作成
```

---

### 4. テスト

**AI に任せること**
- UT（単体テスト）：境界条件・例外系・正常系
- IT（統合テスト）：API〜DB、主要ユースケース、トランザクション・冪等性
- E2E：Playwright でのブラウザ操作スクリプト生成・保守
- テストデータ（fixture / seed）の生成

**テスト戦略（C2 カバレッジ）**

| 層 | 責務 | カバレッジ目標 |
|----|------|--------------|
| UT | 関数・クラス仕様、境界条件、例外 | クリティカル領域：条件網羅（C2）必須 |
| IT | API〜DB、ユースケース、冪等性 | 主要ユースケース全件 |
| E2E | 重要画面のシナリオ | クリティカルパス全件 |

**カバレッジ運用**
- PR でカバレッジが低下した場合はデフォルトブロック（例外は理由必須）
- クリティカル領域（認証・決済・根幹ロジック）は条件網羅（C2）を必須とする
- Flaky テストは即座に修正または quarantine（放置禁止）

**E2E の実運用**

*Step 1：Playwright をインストールする*

```bash
# 新規セットアップ
npm init playwright@latest

# 既存プロジェクトに追加する場合
npm install -D @playwright/test
npx playwright install
```

*Step 2：Codex に E2E スクリプトを生成させる*

```
以下の画面操作に対して Playwright の E2E テストを作成して。

【テスト対象の操作】
{画面の操作手順を番号で記述}

【ベース URL】
{例: http://localhost:3000}

【確認すること】
{テストで検証したい内容}

既存テスト（tests/e2e/）のパターンに合わせた形式で作成すること。
```

*Step 3：CI に追加する（`.github/workflows/ci.yml` に追記）*

```yaml
  e2e:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

*Step 4：失敗時の対処フロー*

1. CI の成果物（スクリーンショット・動画・トレース）をダウンロード
2. トレースをローカルで確認：`npx playwright show-trace trace.zip`
3. Claude Code に原因の切り分けを依頼：
   ```
   Playwright テストが失敗した。原因の仮説を列挙して。
   【失敗したテスト】{テスト名}
   【エラーメッセージ】{エラー全文}
   【スクリーンショット】（貼り付け）
   ```

**どの画面から E2E を始めるか（優先順）**

| 優先度 | 対象 | 理由 |
|--------|------|------|
| 1 | ログイン・認証フロー | 全ユーザーに影響。壊れると全機能が使えなくなる |
| 2 | コアとなるユーザー操作 | 最も使われる機能。リグレッションリスクが高い |
| 3 | 決済・課金フロー | ビジネスクリティカル |
| 4 | 管理画面の主要操作 | 運用に直結する |

---

### 5. レビュー

**AI に任せること**
- PR 差分の要約
- レビューチェックリストの初期埋め
- 静的観点（重複・命名・構造）
- テスト不足箇所の指摘

**人間が必ずやること（→ 詳細は [ai-usage-guidelines.md](ai-usage-guidelines.md)）**
- セキュリティ観点（認証・認可・秘密情報）
- 根幹ロジック（課金・決済・整合性）
- 破壊的変更の確認

---

### 6. リリース

**AI に任せること**
- リリースノート自動生成（PRラベル・タイトルから）
- バージョニング（セマンティックバージョン）
- タグ付けスクリプト

**人間がやること**
- リリース可否の最終判断
- 本番環境への反映承認

**マージ方針**
- squash merge を原則とする（履歴を追いやすい）
- 破壊的変更は移行手順・ロールバック手順を PR に必ず記載

---

### 7. 運用

**AI に任せること**
- エラーログの解析・原因仮説の生成
- インシデント対応手順の下書き
- 定期的な依存関係更新の PR 生成

**人間がやること**
- 監視アラートへの対応判断
- インシデントの最終対応

## PR 粒度・ブランチ戦略

### PR の基本方針

| 原則 | 理由 |
|------|------|
| 1 PR = 1 機能・1 目的 | レビューしやすい。AIの精度も上がる |
| 変更行数は原則 400 行以内 | 超えたら分割を検討 |
| テストを含める | テストのない PR はレビュー対象外 |
| 破壊的変更は移行手順を含める | ロールバック可能な状態でマージ |

### ブランチ戦略（trunk-based 推奨）

```
main（常にデプロイ可能）
  └── feature/issue-{番号}-{概要}  ← 機能追加
  └── fix/issue-{番号}-{概要}      ← バグ修正
  └── refactor/{概要}             ← リファクタ
  └── chore/{概要}                ← 依存更新・CI変更
```

- feature ブランチは main から切って main に戻す
- 長命ブランチは作らない（1週間以内にマージを目安）
- リリースブランチはセマンティックバージョンタグで管理

### 最小変更単位

- バグ修正：原因となる最小箇所のみ修正。ついでのリファクタは別 PR
- 機能追加：UI / ロジック / テスト は同一 PR に含めてよい
- リファクタ：動作変更を含まないこと。テストで担保してから PR
- 依存更新：セキュリティ対応は即 PR。通常更新は週次 or 月次でまとめる

## ドキュメント駆動開発

### 維持するドキュメント

| ドキュメント | 場所 | 更新タイミング |
|------------|------|--------------|
| ADR（アーキテクチャ決定記録） | `docs/adr/` | 技術選定・設計判断時 |
| API 設計書 | `docs/api/` | API 変更時（PR と同時） |
| DB スキーマ | `docs/db/` | スキーマ変更時（PR と同時） |
| README | ルート・各ディレクトリ | 初期構築時・大きな変更時 |
| CHANGELOG | ルート | リリース時（自動生成） |

### AI による生成・更新

- 設計書・ADR の初稿は Claude Code に生成させる
- コード変更時の README 更新も Codex に担当させる
- 「ドキュメントの更新が漏れていたら指摘して」をプロンプトに入れる習慣をつける

## CI/CD 自動化

### PR 作成時に自動実行

```yaml
# 最小構成例（.github/workflows/ci.yml）
- lint（ESLint / Ruff 等）
- typecheck（TypeScript / mypy 等）
- test（UT + IT）
- coverage（低下したらブロック）
- security scan（Trivy / Dependabot / CodeQL）
```

### PR 作成時の自動生成（AI 活用）

- 変更概要・影響範囲のコメント自動付与
- テスト結果サマリの PR コメント
- カバレッジレポートの差分表示

### リリース自動化

```
タグ push → GitHub Actions → リリースノート生成 → GitHub Release 作成
```

## 品質 KPI

| 指標 | 目標（参考値） | 計測方法 |
|------|--------------|---------|
| リードタイム（Issue → マージ） | 3営業日以内 | GitHub Insights |
| PR 平均サイズ | 400行以下 | GitHub Actions で計測 |
| テストカバレッジ（クリティカル領域） | 条件網羅（C2） | カバレッジレポート |
| バグ混入率 | AI導入前と比べて増加させない。四半期ごとに傾向を確認 | Issue ラベル集計 |
| ロールバック率 | 1%以下 | デプロイ記録 |
| MTTR（平均復旧時間） | 2時間以内 | インシデント記録 |
| レビュー滞留時間 | 1営業日以内 | PR 作成〜レビュー開始の差分 |

## 段階導入ロードマップ

### Phase 1：個人利用（1〜2週間）

- Claude Code・Codex CLI を個人の手元で使い始める
- 既存プロジェクトの一部タスクで Issue → PR の流れを試す
- CLAUDE.md の初稿を作る

### Phase 2：チーム試験導入（2〜4週間）

- PR テンプレート・Issue テンプレートを追加
- CODEOWNERS を設定
- CI（lint/test）を整備
- チーム内の一部機能に限定して運用

### Phase 3：標準化（1〜2ヶ月）

- CLAUDE.md をチームで育てる
- カバレッジ・品質 KPI の計測を開始
- E2E 自動化を重要画面から順に追加
- 振り返りでルールを更新

どのフェーズでも、禁止領域への AI 変更は人間レビュー必須。詳細は [ai-usage-guidelines.md](ai-usage-guidelines.md) を参照。
