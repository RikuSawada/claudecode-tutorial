# PR テンプレート

---

## PR の基本方針

- **1 PR = 1 目的**。機能追加・バグ修正・リファクタを混ぜない
- PR を出す前に CI が全緑であることを確認する
- 変更に見合ったテストを含める（テストのない PR はレビュー対象外）
- 破壊的変更（スキーマ変更・API 互換性）は移行手順・ロールバック手順を必ず記載する

---

## テンプレート

以下を `.github/pull_request_template.md` に配置する。

```markdown
## 目的

<!-- この PR で何を達成するか。1〜2行で -->

## 変更点

<!-- 変更した内容を箇条書きで -->
-
-

## 動作確認

<!-- ローカルで確認した内容 -->
- [ ] 正常系：
- [ ] 異常系：
- [ ] 既存機能への影響なし：

## 追加・更新したテスト

<!-- 追加・変更したテストの概要 -->
- UT：
- IT：
- E2E：

## リスクと対策

<!-- この変更で懸念されるリスクと、その対策 -->
- リスク：
- 対策：

## ロールバック手順

<!-- 問題が発生した場合の戻し方 -->

## 人間レビュー必須チェック

<!-- 該当するものにチェック。すべて「なし」の場合も明記する -->
- [ ] 認証・認可の変更
- [ ] 秘密情報・ログへの混入リスク
- [ ] ドメイン根幹ロジック（課金・決済・在庫・計算）
- [ ] スキーマ変更・API 互換性
- [ ] 性能影響（N+1・重いクエリ・非同期化）
- [ ] 監視・アラートへの影響

## 関連 Issue

Closes #{Issue番号}
```

---

## マージ条件

以下をすべて満たすこと。

| 条件 | 確認方法 |
|------|---------|
| CI が全緑 | GitHub Actions のステータス確認 |
| 変更に見合う UT/IT がある | PR の「追加・更新したテスト」欄を確認 |
| カバレッジが低下していない | CI のカバレッジレポートを確認 |
| 人間レビュー必須チェックが完了 | PR テンプレートのチェックボックス確認 |
| 破壊的変更がある場合は移行手順を記載 | PR の「ロールバック手順」欄を確認 |
| CODEOWNERS の承認がある（対象領域の場合） | GitHub のレビュー承認を確認 |

## レビュー運用

### AI レビューの使い方

| 観点 | AI でよい | 人間必須 |
|------|---------|---------|
| 命名・構造・重複 | ◯ | — |
| テストの網羅性の提案 | ◯ | — |
| 差分の要約 | ◯ | — |
| セキュリティの最終判断 | — | ◯ |
| 根幹ロジックの正しさ | — | ◯ |
| アーキテクチャの妥当性 | 参考程度 | ◯ |

### レビューの合意形成

チームで以下を決めてドキュメント化する：

```
AI でレビューしてよい観点：
- コーディングスタイル・命名
- テストのアサーション内容
- 依存関係の循環

人間が必ずレビューする観点：
- 認証・認可ロジック
- 決済・課金ロジック
- DB スキーマ変更
```

### レビュー滞留を防ぐ

- PR 作成から **1営業日以内** にレビュー開始を目標とする
- レビュアーが決まらない PR は担当をアサインする
- 大きな PR（400行超）は分割してから出す

## 具体例：良い PR と悪い PR

### 悪い PR の特徴

```
タイトル：「修正」
変更行数：1200行
テスト：なし
説明：「色々直した」
```

なぜ悪いか：
- 何を確認すればいいかわからない
- テストがないので動作保証がない
- ロールバックの単位が大きすぎる

### 良い PR の特徴

```
タイトル：「feat: 商品検索フィルターに価格帯絞り込みを追加 (#123)」
変更行数：120行
テスト：UT 8件追加（境界値・異常系含む）
説明：目的・変更点・動作確認・リスクが明記されている
```

なぜ良いか：
- 目的が1行でわかる
- テストで動作が保証されている
- リスクが明示されているのでレビュアーが集中できる

## squash merge の運用

```
# PR の squash merge 後のコミットメッセージ形式
{タイプ}: {概要} (#{PR番号})

# タイプ一覧
feat:     機能追加
fix:      バグ修正
refactor: リファクタリング
test:     テスト追加・修正
chore:    依存更新・CI・ドキュメント
```

例：
```
feat: ユーザーメール通知機能を追加 (#42)
fix: ログイン時のセッションタイムアウトが正しく動作しない問題を修正 (#51)
```
