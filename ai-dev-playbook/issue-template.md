# Issue テンプレート

---

## Issue の書き方

### 基本方針

- **1 Issue = 1 機能・1 バグ**。複数の目的を1つの Issue に混ぜない
- 未決事項が残っている Issue は着手しない（未決ラベルを付けて解消を待つ）
- 受入条件（AC）は「これが通れば完成」と言える粒度で書く
- スコープは「やること」だけでなく**「やらないこと」を明記する**

## ラベル運用

| ラベル | 意味 |
|--------|------|
| `feature` | 機能追加 |
| `bug` | バグ修正 |
| `refactor` | リファクタリング（動作変更なし） |
| `chore` | 依存更新・CI・ドキュメント |
| `unresolved` | 未決事項あり。解消するまで着手しない |
| `critical` | セキュリティ・根幹領域。CODEOWNERS でレビュー必須 |
| `blocked` | 別の Issue/PR 待ち |

## タスク分解の指針

### 大きすぎる Issue を分割するタイミング

- 実装に3日以上かかりそう
- PR が 400 行を超えそう
- 複数の機能を同時に変更する

### 分割のパターン

```
# パターン1：機能を分割
[ ] Issue: ユーザー一覧ページの実装
    ├── Sub-Issue: API（GET /users）の実装
    ├── Sub-Issue: 画面コンポーネントの実装
    └── Sub-Issue: E2E テストの実装

# パターン2：段階リリースで分割
[ ] Issue: 検索機能の実装（Phase 1: 商品名のみ）
[ ] Issue: 検索機能の拡張（Phase 2: カテゴリ・価格帯）

# パターン3：設計と実装を分割
[ ] Issue: 決済フローの設計（設計書・ADR の作成）
[ ] Issue: 決済フローの実装（設計承認後に着手）
```

### スコープ管理

- 実装中に「ついで」の変更をしない。別 Issue にする
- フラグ管理（Feature Toggle）が必要な場合は Issue に明記する
- 段階リリースの計画は Issue に記載して関係者に共有する

## Issue 管理の方針（未決）

どこで管理するかはチームで決める。以下は選択肢の整理。

| 方法 | 向いているケース | 弱点 |
|------|----------------|------|
| GitHub Issues（推奨） | エンジニア中心・PR との紐付けが必要 | 非エンジニアが慣れるまで時間がかかる |
| Notion | 長文の要件整理・議事録が多い | PR/CI との連携が弱く二重管理になりがち |
| リポジトリ内 md | 設計書・ADR の管理 | 一覧性・担当・通知が弱い |

**暫定推奨**：実装タスクは GitHub Issues、長文の要件・議事録は Notion、設計書・ADR は repo 内 md。
