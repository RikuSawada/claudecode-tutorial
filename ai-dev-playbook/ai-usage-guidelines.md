# AI 活用ガイドライン

---

## AI に任せる範囲

AIは「実行者」。**決定・判断・責任は人間が持つ**。

| 工程 | AI に任せてよいこと | 人間がやること |
|------|-------------------|--------------|
| 要件定義 | AC 作成、非機能要件洗い出し、曖昧さの指摘 | ビジネス判断・優先順位決定 |
| 設計 | API/DB/画面設計の下書き、ADR 初稿 | アーキテクチャ最終決定、セキュリティ確認 |
| 実装 | コード生成、リファクタ、ドキュメント更新 | 生成物の確認・承認 |
| テスト | UT/IT/E2E コード生成・保守 | テストの妥当性確認 |
| レビュー | 差分要約、静的観点、チェックリスト初期埋め | セキュリティ・根幹のレビュー（必須） |
| リリース | リリースノート生成、タグ付け | リリース可否の最終判断 |

---

## 人間レビュー必須ポイント

### A. セキュリティ（必須・例外なし）

- [ ] 認証・認可（RBAC/ABAC・権限昇格・管理者操作）
- [ ] 入力バリデーション（SQLi / XSS / CSRF / ファイルアップロード）
- [ ] 秘密情報の取り扱い（鍵・トークン・パスワードのハードコード・ログ混入）
- [ ] 依存ライブラリ更新の影響（脆弱性・ライセンス）
- [ ] 外部 API 連携時の認証トークン管理

### B. アプリの根幹（必須・例外なし）

- [ ] ドメインルール（課金・決済・在庫・個人情報・計算ロジック）
- [ ] データ整合性（トランザクション・冪等性・並行実行）
- [ ] 破壊的変更（スキーマ変更・API 互換性・移行手順）

### C. 追加で人間が見るべきポイント（推奨）

- [ ] 境界設計：責務分離（層・モジュール）、依存方向、循環参照
- [ ] 運用：ログの粒度・メトリクス・アラート・ロールバック手順
- [ ] 性能：N+1 クエリ・重いクエリ・キャッシュ・非同期化の必要性
- [ ] テストの妥当性：意味のあるアサーションか（通るだけで検証していないテストでないか）
- [ ] エラーハンドリング：例外の握りつぶし・不適切なエラーメッセージ露出

### D. AI 生成コード特有のリスク

- [ ] コンテキスト不足による仕様ズレ（既存コードと矛盾していないか）
- [ ] 巨大関数・巨大クラスの生成（分割されているか）
- [ ] コピペ的実装（重複・凝集度の低下）
- [ ] ハードコードされた値（URL・ID・設定値）

## AI 用コンテキスト設計

### 「不変コンテキスト」— リポジトリに常駐させる

CLAUDE.md に記載して、毎回 AI に渡す情報。

```
/CLAUDE.md                  ← プロジェクト全体のルール
/docs/ai/
  project_overview.md       ← アーキテクチャ概要・制約
  coding_guidelines.md      ← 命名・例外・ログ・型の規約
  security_guidelines.md    ← 禁止事項・秘密情報の扱い
  testing_guidelines.md     ← UT/IT/E2E の責務・モック方針
```

**CLAUDE.md に必ず書くこと**

```markdown
## プロジェクト概要
（アーキテクチャ・技術スタック・主要コンポーネント）

## ディレクトリ構成
（各ディレクトリの役割）

## コーディング規約
（命名・例外処理・ログの書き方・型）

## テスト方針
（UT/IT/E2E の責務・モックの考え方）

## AIへの禁止事項
（触ってはいけないファイル・領域）

## よく参照するファイル
（エントリポイント・共通ユーティリティ）
```

### 「可変コンテキスト」— Issue/PR ごとに渡す情報

| 項目 | 渡し方 |
|------|--------|
| このIssueの背景・AC・制約 | Issue の本文をプロンプトに貼る |
| 影響範囲（API/DB/画面） | Issue の影響範囲欄から引用 |
| 既存実装の参照先 | 「src/auth/login.ts の実装を参考にして」 |
| 変更してはいけない箇所 | 「src/payment/ は変更しないこと」 |

### 変更してはいけない領域の明示

AI に以下のようにプロンプトで伝える：

```
以下のファイル・ディレクトリは変更禁止：
- src/auth/（認証・認可ロジック）
- src/payment/（決済ロジック）
- .env*, config/production.*（本番設定）
- migrations/（DBマイグレーション。スキーマ変更は別タスクで管理）
```

## セキュリティ・コンプライアンス

### 機密情報の扱い

| 情報 | 禁止行為 |
|------|---------|
| API キー・トークン | ハードコード・プロンプトへの貼り付け |
| パスワード | コードへの埋め込み・ログへの出力 |
| 個人情報 | ログへの出力・AI への貼り付け |
| 本番 DB の接続情報 | リポジトリへのコミット |

**AI へのプロンプトに秘密情報を貼らない。** ダミー値や `<SECRET>` のようなプレースホルダーを使う。

### 依存関係・脆弱性スキャン

- Dependabot（または Renovate）で自動更新 PR を生成
- Trivy / CodeQL / npm audit を CI に組み込む
- セキュリティ対応の更新は即日対応を原則とする

### AI 生成コードのライセンス

- 生成コードの著作権・ライセンスは組織のポリシーに従う
- OSS ライセンスに抵触する可能性のあるコードは法務に確認

## 連携方式

### AI をどこで動かすか

| 場所 | ツール | 用途 |
|------|--------|------|
| ローカル CLI | Claude Code / Codex CLI | 日常的な実装・設計支援 |
| エディタ | Claude in IDE | コーディング中のインラインサポート |
| GitHub Actions | Claude Code（CI 内） | PR 自動レビュー・差分サマリ生成 |

### プロンプトを資産化する

プロンプトは使い捨てにせず、チームの共有資産にする。

```
/docs/prompts/
  requirements.md    ← 要件定義フェーズのプロンプト集
  design.md          ← 設計フェーズのプロンプト集
  implementation.md  ← 実装フェーズのプロンプト集
  review.md          ← レビューフェーズのプロンプト集
  debug.md           ← デバッグのプロンプト集
```

→ 実際のテンプレートは [prompt-template.md](prompt-template.md) を参照

### GitHub 機能の使い方

| 機能 | 用途 |
|------|------|
| Issues | タスク管理の主軸（要件・AC・テスト観点を記載） |
| Projects | スプリント管理・ロードマップ |
| Pull Requests | コードレビュー・CI 結果の確認 |
| Actions | lint/test/deploy の自動化 |
| Discussions | 設計議論・RFC（ADR に昇格させる） |
| Wiki | 運用手順・オンボーディング資料 |

> Notion は長文の要件整理・議事録に使い、実装タスクは GitHub Issues に集約する。
> Issue の代わりに repo 内 md ファイルを使うと一覧性・通知が弱くなるので避ける。

## よくある失敗パターン

| 失敗 | 原因 | 対策 |
|------|------|------|
| AIがドメインルールを無視した実装をする | CLAUDE.md にルールがない | CLAUDE.md にドメイン知識・禁止事項を記載 |
| 生成されたテストが「通るだけ」 | アサーションの内容を確認していない | レビューで「何を検証しているか」を必ず確認 |
| PR が巨大になってレビューできない | Codex に大きすぎるタスクを渡した | タスクを小さく分割してから Codex に渡す |
| secrets がコミットに混入 | AI がファイルを読んで内容を貼り付けた | `.gitignore` + pre-commit hook（git-secrets 等）で防ぐ |
| 根幹ロジックを AI が書き換えた | CODEOWNERS 未設定 | 重要領域に CODEOWNERS を設定し人間レビューを必須化 |
| コンテキスト不足で的外れな実装 | Issue の記載が曖昧 | AC・影響範囲・制約を Issue に必ず書く |
| AI に指摘されたバグを全部直してもらった | 根幹領域の変更を AI に任せた | 根幹領域の変更は AI が提案するだけ・人間が実装 |

## 良い指示と悪い指示

### 要件定義

```
# NG：曖昧
「検索機能を作って」

# OK：具体的なコンテキスト付き
「商品一覧ページに検索フィルターを追加したい。
対象フィールド：商品名・カテゴリ・価格帯。
既存の /api/products のクエリパラメータを拡張する形で実装。
モバイルでも使えること。
この要件でGitHub Issue の下書きを作って（AC・スコープ・テスト観点を含める）。」
```

### 実装依頼

```
# NG：コンテキストなし
「メール送信機能を実装して」

# OK：制約と参照先を明示
「src/notifications/ にメール送信機能を追加して。
- 既存の NotificationService クラスを継承する
- メール送信には現在使っている SendGrid SDK を使う（src/libs/sendgrid.ts 参照）
- エラー時はログを出して例外を上位に投げる（既存の例外処理パターンに従う）
- src/auth/ と src/payment/ は変更しないこと」
```

### レビュー依頼

```
# NG：全部任せる
「このPRをレビューして」

# OK：観点を絞る
「このPRの以下の観点でレビューして：
1. SQLインジェクションのリスクがないか
2. テストのアサーションが意味のある検証をしているか
3. エラーハンドリングが既存パターンと一致しているか
セキュリティの最終判断は自分でするので、懸念点の列挙だけでよい。」
```
